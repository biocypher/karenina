# SPEC-007: Testing and Feedback Harness

**Status:** Draft for review
**Charter:** [README.md](README.md) §3, SPEC-007
**Principles:** P8 (self-verifying), P4
**Destination:** `tests/` reorganization, `Makefile` targets, CI workflows, the cull applied via Phase 0 Track C
**Decisions inherited:** Three tiers. Audit-then-cull. Contracts on PR, goldens on merge, live manual. Live coverage: all adapters, Codon vLLM, z.ai GLM, Anthropic, at least one agentic path, abstract cases from the example gamma.

---

## 1. Tier 1: Contract and Parity Tests

Deterministic, fast, run on every PR. The named set (each name is the actual test module):

| Test | Asserts | Spec it enforces |
|---|---|---|
| `tests/parity/test_config_completeness.py` | Every core config field path has a registered parity case | SPEC-004 §3.1 |
| `tests/parity/cases/` (per sub-config) | Each field observably changes behavior | SPEC-004 §3.2 |
| `tests/parity/test_path_parity.py` | The same question evaluated via the QA projection, the TaskEval projection (pre-fed answer), and a single-node scenario yields equivalent verdicts and result schemas | SPEC-003 §1.3, P5 |
| `tests/contracts/test_resume_roundtrip.py` | Run, interrupt after k results, resume: no loss, no duplicates, transient retried, stuck skipped, summary reports it | SPEC-006 §1 |
| `tests/contracts/test_teardown_leak.py` | After `Driver.run` exits, the client debug registry is empty | SPEC-003 §5 |
| `tests/contracts/test_sink_conformance.py` | Every sink implementation against protocol v2 semantics (single-consumer serialization assumed, streaming `on_start`) | SPEC-006 §2 |
| `tests/conformance/<port>/` | Every registered adapter against its port: gateway usage (no out-of-gateway provider calls), concurrency granularity honored, capability declarations truthful | SPEC-003 §3, P2 |
| `tests/conformance/test_error_battery.py` | Every adapter maps simulated wire failures (timeout, 429, 401, malformed body) to identical `FailureClass`es | SPEC-005 §1 |

Conformance suites run against any adapter discovered via entry points, so third-party adapters get the same verification (P8).

## 2. Tier 2: Recorded Goldens

Run on merge to the main development branch.

- **Capture:** fixtures come exclusively from real pipeline runs via `scripts/capture_fixtures.py` (existing discipline, restated as binding). Hand-written fixtures are a review reject.
- **Coverage:** one golden per projection (QA, scenario, TaskEval) × one representative config per sub-config region (defaults, deep judgment on, rubric on, guards on). Assertions target the final `VerificationResult` and `RunSummary` content, never intermediate calls.
- **Update policy:** when behavior legitimately changes, goldens are regenerated by the capture script and the diff is reviewed in the PR like code. Editing a golden by hand or loosening an assertion to pass is a review reject.

## 3. Tier 3: Live Scoreboard

Manual, run between phases, before releases, and after provider-side changes. Interface: `make engine-health` (wraps `pytest -m live`), printing a per-case pass/fail board and writing a JSON report.

Case list (each is an example-gamma case, SPEC-008 §5, run live):

| Case | Path exercised | Provider |
|---|---|---|
| `qa_basic` | QA projection, default pipeline | Codon vLLM (volume), z.ai GLM |
| `qa_judged` | Separate judge, deep judgment | Anthropic |
| `qa_agentic` | Agent adapter answering (claude_agent_sdk), `"call"` granularity | Anthropic |
| `scenario_short` | 2-turn scenario, turn tasks, dynamic spawn | z.ai GLM |
| `taskeval_logs` | TaskEval projection on pre-collected outputs | judge only |
| `resume_drill` | Kill `qa_basic` mid-run, resume, verify §SPEC-006 semantics live | Codon vLLM |
| `concurrency_drill` | Caps under load: assert logged max in-flight ≤ configured cap, global and per-model | Codon vLLM |
| `rubric_traits` | Rubric evaluation incl. one agentic trait | Anthropic |

Every registered adapter appears in at least one case. Budget: a full board targets under one dollar of API spend and under 15 minutes wall-clock (vLLM carries the volume). Flake policy: one automatic retry per case, a case that flakes twice in a row is filed as a finding, not retried into silence (P4).

## 4. The Cull Rubric (applied by Phase 0 Track C)

Classify each test file, or coherent test class, by answering three questions in order:

1. **Does it assert observable behavior at a public or port boundary** (facade, CLI, port protocol, result schema, on-disk format)? → **contract: keep** (and migrate to the tier-1 layout if it belongs there).
2. **Does it pin behavior an audit finding says is wrong** (cite the finding id)? → **bug-characterization: delete in the same PR as the fix**, replaced by the test of the correct behavior.
3. **Does it mock internals of the unit under test, assert call sequences, or reach into private attributes?** → **implementation-mirror: delete in the PR that refactors its target.** It would fail on any restructure and catches no real regression.

Files matching none (for example pure schema-validation unit tests) default to keep. Ambiguous files are flagged in the findings record per SPEC-009 §4 rule 7. The classification lands in `specs/audit/findings-tests.md`, and deletions always travel with the PR that makes them safe, never as a standalone purge.

## 5. CI Gating

| Trigger | What runs |
|---|---|
| Every PR | Tier 1 (contracts, parity, conformance) plus the surviving unit suite. Target: under 5 minutes |
| Merge to dev branch | Tier 1 + Tier 2 goldens |
| Manual (`make engine-health`) | Tier 3, requires provider keys and the vLLM server, run between phases and before releases |
| Release | All three tiers green is a release precondition |

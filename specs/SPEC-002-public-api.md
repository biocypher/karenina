## 1. The Five-Line Happy Path

The canonical tier-1 experience, exactly as it will appear at the top of the README and quickstart:

```python
from karenina import Benchmark

b = Benchmark.from_csv("questions.csv")
results = b.run("anthropic/claude-haiku-4-5")
print(results.summary())
```

Named defaults behind it (each is a binding commitment, tested by the example gamma):

| Default | Value | Visibility |
|---|---|---|
| Judge (parsing model) | Same as the answering model | `UserWarning` once per run plus a run-summary caveat: minimal config works with one key, but self-judging is never silent |
| Templates | Questions without a template get one auto-generated by the judge model at the start of `run()` | Warning with the count, generated templates saved back to the benchmark |
| Adapter | `langchain`, model strings parsed as `"provider/model"` | Documented |
| Pipeline | validate, generate, parse, verify, finalize. Trace-sanity guards (recursion, trace validation) on, optional guards (abstention, sufficiency) off | Documented defaults table in the config reference |
| Sink | In-memory. `b.run(..., save="run.jsonl")` switches to the progressive file sink with resume | Documented |
| Concurrency | `ExecutionTuning()` defaults (SPEC-003 §3) | Documented |

## 2. Core Nouns and the Verb Grammar

Tier-1 nouns: `Benchmark`, `Results`. Tier-2 nouns: `Template`, `Rubric`, `Scenario`, the SPEC-004 config objects, `TaskEval`. Experimental: `karenina.experimental.engine` (SPEC-003 §2).

The uniform verb grammar, binding for all nouns:

| Verb | Meaning | Rule |
|---|---|---|
| `from_*(...)` | Construct from a source (`from_csv`, `from_dict`, `from_checkpoint`) | Classmethod, returns the noun. Every noun with a `save` has at least one `from_*` that round-trips it |
| `add(...)` / `add_*(...)` | Grow the object in place | Returns `self` for chaining |
| `run(...)` / `arun(...)` | Execute, returning `Results` | Sync facade and async twin, identical signatures (SPEC-003 §4) |
| `resume(...)` | Continue an interrupted run from its save path | Applies SPEC-006 §1 semantics |
| `extend_*(...)` | Re-run downstream stages on captured traces | Raises `ReplayMissError` without capture (SPEC-006 §3) |
| `save(path)` / `export(format)` | Persist / produce interchange output | `save` round-trips, `export` is one-way |
| `summary()` / `to_frame()` | Human-oriented digest / DataFrame | On `Results` and other reportable nouns |

Signature conventions (the enforceable consistency contract):

1. First positional argument is the subject (source path, model string). Everything else is keyword-only.
2. No boolean positional arguments, ever.
3. Model parameters accept a string (`"provider/model"`) or a `ModelConfig` everywhere a model is named. Strings are sugar for default-config models, the upgrade path is in place, not a different parameter.
4. `get_*` / `create_*` / `build_*` prefixes keep their existing meanings (cached-or-new / always-new / config-dict).
5. New public API must reuse these verbs before inventing one. A new verb requires an ADR.

## 3. Progressive Disclosure: One Task, Four Rungs

The same evaluation at increasing depth. Each rung keeps every prior argument valid: you only add, never migrate.

```python
# Rung 0: defaults
results = b.run("anthropic/claude-haiku-4-5")

# Rung 1: named preset (shareable, versioned config bundles)
results = b.run("anthropic/claude-haiku-4-5", preset="strict-judging")

# Rung 2: explicit config objects (SPEC-004 sub-configs)
results = b.run(
    "anthropic/claude-haiku-4-5",
    config=VerificationConfig(
        judging=JudgingConfig(model="anthropic/claude-opus-4-8", deep_judgment=True),
        guards=GuardsConfig(abstention_check=True),
    ),
)

# Rung 3: custom ports (entry-point registered adapter, custom traits)
results = b.run(
    ModelConfig(adapter="my-adapter", model="local-model"),
    config=VerificationConfig(rubric=RubricConfig(traits=[my_callable_trait])),
)
```

Presets are serialized `VerificationConfig` compositions (rung 2 objects), so the ladder is one representation throughout. A preset plus explicit `config` merges with explicit winning, and the merge is reported in the run summary.

## 4. Deliberately Not in Tier 1

Template and rubric authoring (auto-generation covers tier 1, authoring is tier-2 with skills), scenario graph construction, the engine/driver surface, sub-config internals, adapter registration, deep judgment tuning, MCP configuration. Tier-1 docs do not mention these beyond a single "going deeper" pointer.

## 5. Backwards Mapping (delete over deprecate)

The full old-name → new-name table is produced by applying this spec during Phase 4 and recorded in the migrating ADR. Known commitments now:

| Current | Fate |
|---|---|
| `Benchmark.run_verification(...)` | Becomes `run(...)` (and `arun`). Old name deleted, no alias |
| `resume_verification(...)` | Becomes `resume(...)` |
| `extend_template` / `extend_rubric` | Names survive, signatures normalized to the grammar |
| ~58-parameter runner surface | Absorbed by `TaskSpec` (SPEC-004 §4), never public |
| CLI `karenina verify` | Survives, flags re-derived from the sub-config schema so CLI and API stay isomorphic |
